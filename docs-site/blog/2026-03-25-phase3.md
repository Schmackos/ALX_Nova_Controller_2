ALX Nova PRs #80–83 land the Phase 1+2 hardening work — four releases that collectively close the most significant reliability and correctness gaps identified in the foundation analysis. Here is what changed and what it means for devices in the field.

<!-- truncate -->

## What Changed

### PR #80 — WebSocket Protocol Version + JSON Envelope Helper

Every initial WebSocket broadcast now includes a `protocolVersion` field. Clients that depend on specific broadcast shapes can check this field and warn the user if the firmware is newer or older than expected.

A new `json_response()` helper standardises the REST response envelope. All new endpoints use `{"ok": true, "data": {...}}` for success and `{"ok": false, "error": "..."}` for failure, giving API consumers a consistent shape to pattern-match against.

### PR #81 — HAL Fault Persistence (NVS)

HAL device faults are now persisted to NVS and survive reboots. If a device enters the Error state and the device is not physically present when the firmware restarts, the fault record is preserved and surfaced via `GET /api/hal/faults`. This makes diagnosis of intermittent hardware failures dramatically easier — the fault is no longer lost the moment the device reboots.

Faults can be cleared explicitly via `DELETE /api/hal/faults/:slot` or `DELETE /api/hal/faults` (all faults).

### PR #82 — Pipeline Latency Measurement

The audio pipeline now measures and reports end-to-end latency in microseconds. `GET /api/pipeline/status` includes a `timing` object with `totalE2eUs`, `dspUs`, and `matrixUs` fields. This gives a concrete, measurable baseline for DSP optimisation work and makes it straightforward to verify that a new DSP stage does not push latency past acceptable bounds.

### PR #83 — Lock-Free Atomic Sink/Source Operations + HAL Power Management

**Atomic pipeline operations:** The previous implementation used `vTaskSuspendAll()` to protect sink and source list mutations. This has been replaced with an `__ATOMIC_RELEASE` / `__ATOMIC_ACQUIRE` sentinel pattern — sink and source pointer swaps are now lock-free. `ScopedSchedulerSuspend` is removed. The audio task on Core 1 is no longer blocked during HAL init/deinit operations.

**HAL power management:** Devices that report `HAL_CAP_POWER_MGMT` (capability bit 13) now participate in a three-state power model: `ACTIVE`, `STANDBY`, and `OFF`. Power state is controllable via `POST /api/hal/devices/power` and reflected in the WebSocket broadcast and web UI device cards. The `HalPowerState` enum is defined in `src/hal/hal_types.h`.

## Upgrade Instructions

Update via the web dashboard: **Settings > OTA > Check for Updates**. The update applies automatically and reboots the device. No configuration changes are required after the update.

If you prefer manual flashing, download the `.bin` from the [GitHub Releases page](https://github.com/Schmackos/ALX_Nova_Controller_2/releases) and upload it via the Manual Upload area in the OTA panel.

## Breaking Changes

None. All REST and WebSocket API changes in this release are additive — new fields are added, no existing fields are removed or renamed. Existing clients will continue to work without modification.

## What Is Next

The roadmap for Phase 3 includes:

- **Hardware ASRC** — polyphase sample rate conversion engine for cross-domain mixing (tracking in `feature/phase2-asrc-dsd`)
- **DSD passthrough** — native DSD/DoP routing through the matrix to DACs that report `HAL_CAP_DSD`
- **ES module build** — migrating `web_src/` to proper ES modules with a bundler step
- **Loadable plugins** — HAL driver loading from LittleFS at runtime, without a firmware reflash

Follow progress on [GitHub](https://github.com/Schmackos/ALX_Nova_Controller_2).
